# Архитектура E-Waste Passport

Этот документ описывает архитектурные решения, структуру компонентов и принципы, которыми руководствуется проект.

---

## Обзор системы

E-Waste Passport построен как классическое трёхзвенное веб-приложение с явным разделением на слои: API-бекенд (Django), SPA-фронтенд (Vue.js) и реляционная база данных (PostgreSQL). Дополнительно — асинхронная очередь задач (Celery + Redis) для уведомлений и генерации отчётов.

```
┌────────────────────────────────────────────────────────────┐
│                        Клиенты                             │
│   Browser (Vue SPA)   Mobile (PWA)   External API Client   │
└──────────────┬─────────────┬────────────────┬─────────────┘
               │             │                │
               ▼             ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                    Django REST Framework                     │
│              /api/v1/  —  JWT Auth  —  Swagger UI           │
├──────────────┬──────────────┬──────────────┬────────────────┤
│   devices/   │manufacturers/│  recyclers/  │    users/      │
│   app        │   app        │   app        │    app         │
├──────────────┴──────────────┴──────────────┴────────────────┤
│                     Django ORM                               │
└──────────────────────┬──────────────────────────────────────┘
                       │
          ┌────────────┴────────────┐
          ▼                         ▼
   PostgreSQL 16               Redis 7
   + PostGIS                (кэш + брокер)

                         (уведомления, отчёты)
                                    │
                                    ▼
                          S3 / MinIO (файлы)
```

## Компоненты бекенда

### Django-приложения

Бекенд разбит на независимые Django-приложения по доменному принципу. Каждое приложение отвечает за свою область и не должно импортировать модели напрямую из другого приложения — взаимодействие через сервисный слой или сигналы.

#### `devices`

Центральный модуль. Содержит основные сущности системы.

Модели:
- `Device` — карточка модели устройства (производитель, категория, материальный состав в JSON, экологический индекс)
- `DeviceUnit` — конкретный экземпляр устройства с серийным номером и QR-кодом
- `DevicePassport` — паспорт экземпляра: текущий владелец, статус жизненного цикла, история переходов
- `DisassemblyGuide` — пошаговые инструкции разборки, привязанные к модели

Сервисы:
- `QRService` — генерация уникального QR-кода и публичной ссылки на паспорт
- `PassportService` — управление переходами статуса (`MANUFACTURED → SOLD → IN_USE → END_OF_LIFE → RECYCLED`)

#### `manufacturers`

Управление аккаунтами производителей. Производитель — организация, которая регистрирует модели устройств и загружает партии серийных номеров.

Модели:
- `Manufacturer` — профиль организации, API-ключ, WEEE-регистрационный номер
- `DeviceBatch` — партия зарегистрированных устройств с датой загрузки

Функционал: bulk-импорт партий через CSV или JSON, генерация ZIP-архива с QR-кодами для печати.

#### `recyclers`

Переработчики и пункты приёма.

Модели:
- `Recycler` — организация-переработчик с сертификатами
- `CollectionPoint` — физический пункт приёма (геоточка PostGIS, часы работы, принимаемые категории)
- `RecyclingRecord` — факт приёмки: какой `DeviceUnit`, кем принят, дата

Сервисы:
- `GeoService` — поиск ближайших `CollectionPoint` по координатам (PostGIS `ST_DWithin`)

#### `users`

Владельцы устройств. Опциональная регистрация — паспорт работает и без неё.

Модели:
- `UserProfile` — расширение стандартного `User`, настройки уведомлений
- `DeviceOwnership` — связь пользователь–устройство с датами владения

#### `notifications`

Celery-задачи для асинхронной работы.

Задачи:
- `send_end_of_life_reminders` — ежемесячный cron, находит устройства старше N лет и отправляет напоминания владельцам
- `generate_weee_report` — генерация PDF-отчёта для производителя за период
- `notify_collection_point_opened` — алерт владельцам в радиусе при появлении нового пункта приёма

### Сервисный слой

Бизнес-логика вынесена из views в отдельный слой `services.py` внутри каждого приложения. Views занимаются только сериализацией запроса/ответа и вызовом сервисов. Это упрощает тестирование и переиспользование логики.

```python
# Пример: views.py вызывает сервис, не содержит логики
class DeviceUnitActivateView(APIView):
    def post(self, request, serial_number):
        unit = get_object_or_404(DeviceUnit, serial_number=serial_number)
        passport = PassportService.activate(unit, owner=request.user)
        return Response(PassportSerializer(passport).data)
```

### Аутентификация и авторизация

Используется два механизма в зависимости от типа клиента:

- **JWT (djangorestframework-simplejwt)** — для фронтенда и мобильного PWA. Access token 15 минут, refresh token 30 дней.
- **API Key** — для производителей, интегрирующих регистрацию устройств в свои ERP-системы. Ключ передаётся в заголовке `X-API-Key`.

Разграничение прав через Django permissions + кастомные `IsManufacturer`, `IsRecycler` permission-классы.

### Публичные страницы паспорта

Страница паспорта, открываемая по QR-ссылке (`/passport/<uuid>/`), намеренно сделана публичной и не требует авторизации. Она рендерится на стороне Django как server-side rendered HTML (Django Templates) — это обеспечивает мгновенную загрузку при сканировании QR и корректный SEO/OG-превью.

Остальной фронтенд — SPA на Vue.js.

## Компоненты фронтенда

Vue.js SPA разбит на три логических раздела, каждый со своим набором страниц и Pinia-стором.

### Разделы

**Публичный портал** (`/`) — лендинг, страница паспорта по QR, карта пунктов приёма. Доступен без авторизации.

**Личный кабинет владельца** (`/my/`) — список зарегистрированных устройств, напоминания, история утилизации.

**B2B-панель** (`/dashboard/`) — для производителей и переработчиков. Загрузка партий, аналитика, управление пунктами приёма.

### Ключевые технические решения

- **Pinia** вместо Vuex — более простой и типобезопасный стейт-менеджмент
- **Vue Router** с lazy-loading маршрутов — B2B-панель грузится только при входе
- **Axios instance** с перехватчиком — автоматически подставляет JWT и обновляет токен при 401
- **Leaflet + vue-leaflet** — карта пунктов приёма без зависимости от Google Maps API на фронтенде

## База данных

### Схема (ключевые таблицы)

```
manufacturers
  id, name, weee_number, api_key_hash, created_at

devices (модели устройств)
  id, manufacturer_id, name, category, material_composition (jsonb),
  eco_index, created_at

device_units (экземпляры)
  id, device_id, serial_number, qr_uuid (uuid, unique), manufactured_at

device_passports
  id, unit_id (1:1), owner_id (nullable), status, activated_at, updated_at

collection_points
  id, recycler_id, name, location (geometry POINT), address,
  accepted_categories (array), working_hours (jsonb), is_active

recycling_records
  id, unit_id, collection_point_id, recycled_at, notes
```

### Решения по данным

**`material_composition` как JSONB** — состав устройства варьируется в зависимости от категории (смартфон vs холодильник vs ноутбук), поэтому структура хранится в гибком JSONB, а не в отдельных столбцах. Для поиска по составу используются GIN-индексы.

**PostGIS для геопоиска** — поиск ближайших пунктов приёма использует нативные геопространственные индексы PostgreSQL, что на порядок быстрее, чем вычисление расстояний на уровне приложения.

**UUID для публичных идентификаторов** — `qr_uuid` в `device_units` — это UUID, который идёт в QR-код и публичную ссылку. Внутренние `id` (integer) не экспонируются наружу.

## Жизненный цикл устройства

```
MANUFACTURED ──► SOLD ──► IN_USE ──► END_OF_LIFE ──► RECYCLED
     │                       │                           │
     │   (производитель       │   (владелец или           │   (переработчик
     │    регистрирует)       │    система по возрасту)   │    подтверждает)
     │                       │                           │
     └── QR генерируется      └── напоминания             └── RecyclingRecord
         при создании             активируются                создаётся
```

Переходы статусов строго последовательны и управляются `PassportService`. Обратный переход невозможен (за исключением административной коррекции).

## Масштабирование

На текущем этапе архитектура рассчитана на вертикальное масштабирование (один сервер, более мощная машина). При росте нагрузки предусмотрены следующие шаги:

**Горизонтальное масштабирование бекенда** — Django stateless, несколько инстансов за Nginx/load balancer. Session state в Redis.

**Read replica PostgreSQL** — аналитические запросы (дашборды) перенаправляются на реплику, не нагружая основную БД.

**CDN для статики и файлов** — PDF-инструкции и QR-изображения раздаются через CDN (CloudFront/Cloudflare), снимая нагрузку с приложения.

**Celery с несколькими воркерами** — задачи генерации отчётов и уведомлений горизонтально масштабируются добавлением воркеров.

## Решения, которые были приняты сознательно

**Django вместо FastAPI** — Django выбран из-за встроенного ORM, admin-панели (критична для управления данными производителей на старте) и зрелой экосистемы. FastAPI быстрее в throughput, но разница несущественна при данных нагрузках.

**Vue.js вместо React** — более пологая кривая обучения для команды; Composition API покрывает все потребности проекта.

**PostgreSQL вместо MongoDB** — несмотря на наличие JSONB-полей, основная схема реляционна. JSONB в PostgreSQL даёт гибкость там, где она нужна, без отказа от транзакций и joins.

**Server-side render для страниц паспорта** — страница, открываемая по QR, должна грузиться мгновенно, в том числе на медленных мобильных соединениях. SSR решает эту задачу без отдельного Next.js/Nuxt.
